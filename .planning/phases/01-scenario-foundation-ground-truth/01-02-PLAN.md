This task implements GT-01 (Opus generates gold-standard outputs), GT-02 (individual JSON files), and GT-04 (consistency check).

**Ground truth generation flow (per RESEARCH.md Steps 1-4):**

**Step 1 -- Validate inputs:** Read all 9 input files and confirm they have the required fields per input_schema (company_name required, company_description required). Flag if all inputs are same industry/size (per Pitfall 2).

**Step 2 -- Pass 1:** For each of the 9 inputs, generate expected_output with these 6 fields:
- `industry`: Casual industry label per D-13. "commercial cleaning" not "janitorial services". Use label_definitions from scenario.json.
- `company_size`: One of ["startup", "SMB", "mid-market", "enterprise"] per label_definitions (startup: pre-revenue or <20 employees; SMB: 20-200; mid-market: 200-2000; enterprise: 2000+).
- `decision_makers`: Array of casual titles. "CMOs" not "Chief Marketing Officers", "ops directors" not "Directors of Operations".
- `pain_points`: Array of 2-4 pain points SPECIFIC to this company's description. Reference details from their input. Not generic.
- `icp_fit`: One of ["strong", "moderate", "weak"]. At least 1-2 companies MUST be "weak" (likely abacus-ai and/or title-one per Pitfall 2).
- `reasoning`: One sentence explaining why this company is strong/moderate/weak fit.

**Step 3 -- Pass 2:** Generate expected_output for all 9 inputs AGAIN, independently. Do NOT look at Pass 1 results while generating Pass 2. This is the consistency check per D-08.

**Step 4 -- Compare:** For each input, compare Pass 1 and Pass 2 field by field:
- Phrasing differences are OK (agreement): "CMOs" vs "marketing leaders" = agreement
- Material differences are NOT OK (disagreement): "mid-market" vs "SMB" = disagreement, "strong" vs "moderate" = disagreement
- Record disagreements in the consistency_check.disagreements array with both values
- Use a simple content hash (first 6 chars of a hash of the JSON-stringified expected_output) for pass_1_hash and pass_2_hash

**Write ground truth files:** Create one JSON file per input in ground-truth/. Use the exact structure from the interfaces block above. Set review.status to "pending" for all files. Leave the "split" field as "pending" (assigned after review).

When disagreements exist between Pass 1 and Pass 2, use your best judgment to pick the correct value as the expected_output, and record the disagreement with both values and your rationale. The human review step will validate these choices.
