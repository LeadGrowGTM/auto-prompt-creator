  <action>
For each of the 9 inputs in v001-raw.json, run the Opus judge 3 times (per D-04, D-05, D-06, D-07):

**Skip condition:** If `parse_failure: true` for an input, do NOT run the judge. Set all dimension scores to 0, all normalized scores to 0.0, weighted_score to 0.0. Mark `judge_skipped: true, judge_skip_reason: "parse_failure"`.

**For each non-parse-failure input:**

1. Read the corresponding ground truth file to get `expected_output`.
2. Read the `haiku_output` from v001-raw.json.
3. Construct the judge prompt with this exact structure (per D-05, EVAL-03):

```
You are evaluating a model's ICP classification output against ground truth.

## Ground Truth (the correct answer)
[paste expected_output JSON from ground truth file]

## Model Output (what the model produced)
[paste haiku_output JSON from v001-raw.json]

## Rubric Dimensions

### 1. Accuracy (weight: 0.4)
Fields are factually correct given the input. Industry matches, size is right, decision makers are plausible.
5/5 anchor: For DBS Building Solutions (340 emp, $28M), output says industry="commercial cleaning", company_size="mid-market". Both factually correct.
1/5 anchor: For DBS Building Solutions, output says industry="technology", company_size="startup". Completely wrong given the input.

### 2. Label Normalization (weight: 0.25)
Uses casual, normalized business language. No corporate jargon, no full title expansions, no formal industry taxonomy.
5/5 anchor: decision_makers=["ops directors", "facility managers"], industry="commercial cleaning"
1/5 anchor: decision_makers=["Chief Operating Officers", "Facility Management Directors"], industry="janitorial services and facility management solutions"

### 3. Completeness (weight: 0.2)
All required fields present with substantive values. No 'N/A' or placeholder answers.
5/5 anchor: All 6 fields present with substantive values. Arrays have 2+ items.
1/5 anchor: Missing icp_fit field, pain_points=["N/A"], reasoning=""

### 4. Specificity (weight: 0.15)
Pain points and reasoning are specific to this company, not generic filler.
5/5 anchor: pain_points=["scaling QA across corporate, medical, and industrial site types", "ServiceTitan dispatch limitations as they grow across the Midwest"]
1/5 anchor: pain_points=["needs better marketing", "wants to grow revenue", "looking for efficiency improvements"]

## Instructions
Score each dimension 1-5. Provide a brief rationale per dimension (1-2 sentences).
Return ONLY a JSON object, no other text:
{
  "accuracy": { "score": N, "rationale": "..." },
  "label_normalization": { "score": N, "rationale": "..." },
  "completeness": { "score": N, "rationale": "..." },
  "specificity": { "score": N, "rationale": "..." }
}
```

4. Run this judge prompt 3 times (3 independent calls in the same session). Parse each response as JSON.
5. For each dimension, collect the 3 scores, sort them, take the median (middle value) per D-06.
6. Compute normalized scores: `normalized = (median_score - 1) / 4` (maps 1->0.0, 5->1.0).
7. Compute weighted input score: `weighted_score = sum(normalized_d * weight_d)` where weights are accuracy=0.4, label_normalization=0.25, completeness=0.2, specificity=0.15 (per D-08).
8. Store the median rationale (from the judge run whose scores are closest to the medians).

After all 9 inputs are judged, compute aggregates (per D-09, D-10):
- `train_score` = arithmetic mean of weighted_score for inputs where split="train" (5 inputs)
- `validation_score` = arithmetic mean of weighted_score for inputs where split="validation" (4 inputs)
- `overall_score` = arithmetic mean of all 9 weighted_scores
- `dimension_averages` = mean of each normalized dimension across all 9 inputs

IMPORTANT SCORING NOTES:
- Use `(score - 1) / 4` normalization, NOT `score / 5`. A judge score of 1 = 0.0, score of 3 = 0.5, score of 5 = 1.0.
- Weights MUST sum to 1.0: 0.4 + 0.25 + 0.2 + 0.15 = 1.0
- If a judge run returns unparseable JSON, retry once. If retry also fails, use scores from the other 2 runs (take mean instead of median for that input).
